# Generic & Config-Driven Item Type System

## Overview

Modernization effort to make the Centy daemon fully generic and configurable. Rather than hard-coding special logic for `issue` and `doc` types, all item types are defined by a `.centy/<type>/config.yaml` file that declares their capabilities (display numbers, status, priority, soft-delete, assets, org sync, move, duplicate), statuses, priority levels, and custom fields. The daemon's storage, gRPC API, and business logic are all routed through a single generic code path.

## Implementation notes

All item types (built-in and custom) use the same code path through the generic item type system. Config.yaml is the single source of truth for item type capabilities. The registry is built at daemon startup by scanning `.centy/` directories. Backward compatibility is maintained during the transition: gRPC accepts both the old `ItemType` enum (v1 clients) and new string `item_type` field. A one-time migration inserts missing config.yaml files into existing projects.
