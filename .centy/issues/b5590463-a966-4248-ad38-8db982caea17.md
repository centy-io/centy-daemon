# lint: enable clippy::indexing_slicing to prevent index-out-of-bounds panics

Direct array/slice indexing (collection\[i\], &slice\[start..end\]) panics at runtime if the index is out of bounds. There are 20+ instances of direct indexing in production code that could panic if assumptions about input are violated.

Examples in the codebase:

* src/search/parser.rs:219 — &raw\[1..raw.len() - 1\] (assumes raw.len() >= 2)
* src/item/entities/pr/crud.rs:936-938 — lines\[0\] (guarded by is_empty() but not enforced by type system)
* src/item/entities/pr/remote.rs:71-72 — parts\[0\], parts\[1\] (guarded by parts.len() >= 2)
* src/common/frontmatter.rs:50,55,64 — multiple lines\[idx..\] slicing operations
* src/item/entities/issue/id.rs:58 — &id\[..8\] (assumes id length >= 8)
* src/item/entities/pr/id.rs:52 — &id\[..8\]
* src/workspace/path.rs:53,110,125 — multiple string slicing operations
* src/metrics.rs:46 — uuid::Uuid::new_v4().to_string()\[..8\]

While many have runtime guards before the access, the indexing_slicing lint would enforce the use of safe alternatives like .get(), .first(), .split_at(), or .get(start..end) which return Option instead of panicking.

Proposal: Add indexing_slicing = “warn” to Cargo.toml \[lints.clippy\], then promote to deny in main.rs/lib.rs for non-test code (same pattern as unwrap_used). Replace direct indexing with safe alternatives like .first().ok_or_else(|| Error::EmptyInput)?.
