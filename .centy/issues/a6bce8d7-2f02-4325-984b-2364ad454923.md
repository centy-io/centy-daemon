# Customization framework: custom item types, templates, commands, and configurable settings

This issue consolidates the full customization framework for Centy, covering every dimension where users and teams need to adapt the tool to their own workflows and conventions. It merges the original issues #135, #136, #138, #139, and #140.

## Overview

Centy ships with a fixed set of assumptions: a hardcoded `.centy` folder name, a small set of built-in item types, two hardcoded editor integrations, generic templates, and no concept of per-type custom commands. The customization framework removes all of these constraints, turning Centy into a flexible local project management framework that teams can shape to fit their conventions rather than the other way around.

The foundation of this framework is the **per-item-type `config.yaml`** (see #135 design). Each item type lives in its own subfolder under the project folder and is fully self-contained — its configuration, templates, and items are all co-located. This one structural decision enables all the other customizations below.

## Checklist

### Custom item types (#135)

- [ ] Define `config.yaml` schema for item types (display number, statuses, priority levels, features, template reference)
- [ ] Implement item type registry that discovers types by scanning `<project-folder>/*/config.yaml` at daemon startup — no central registry file
- [ ] Update `centy init` to create `issues/` and `docs/` folders with their default `config.yaml` files
- [ ] Add migration (`centy update`) to insert `config.yaml` into existing projects that predate this change
- [ ] Replace hardcoded `ItemType` enum with dynamic registry lookups throughout the daemon

#### `config.yaml` example

```yaml
name: Issue
features:
  displayNumber: true
  status: true
  priority: true
  softDelete: true
  assets: true
  orgSync: true
statuses:
  - open
  - in-progress
  - closed
defaultStatus: open
priorityLevels: 3
template: templates/default.md
```

- [ ] Allow users to configure the project folder name globally (`~/.centy/config.json`) or per-project — default remains `.centy` for backward compatibility
- [ ] Auto-detect and respect existing `.centy` folders in projects created before this change (no manual migration needed)
- [ ] Replace all hardcoded `.centy` path references in the daemon, CLI, and integrations with the configured name
- [ ] Ensure documentation, error messages, and logs reference the configured name dynamically

### Configurable editor support (#138)

- [ ] Replace the hardcoded `EditorType` enum and editor-specific RPCs with a config-driven editor registry
- [ ] Define editor config schema: `id`, `name`, `open_dir`, `open_file`, `detect`, `terminal_wrapper`, `setup_workspace`
- [ ] Re-express built-in VS Code and Terminal implementations as default editor config entries (same code path as custom editors)
- [ ] Users can define editors in `~/.centy/config.json`; projects can select a default via `default_editor` in `.centy/config.json`
- [ ] Unify editor-specific RPCs (`OpenInTempVscode`, `OpenInTempTerminal`, etc.) into single `OpenInTempWorkspace` and `OpenStandaloneWorkspace` RPCs that accept an editor ID string
- [ ] Keep old editor-specific RPCs temporarily as thin wrappers during transition period, then deprecate

#### Editor config schema example

```json
{
  "id": "zed",
  "name": "Zed",
  "open_dir": "zed {dir}",
  "open_file": "zed {file}:{line}",
  "detect": "which zed",
  "terminal_wrapper": false
}
```

### Custom templates per item type (#139)

- [ ] Templates live inside each item type's own folder under a `templates/` subdirectory
- [ ] `config.yaml` declares the default template and lists all available named templates
- [ ] `centy init` creates default template files for built-in `issues/` and `docs/` types
- [ ] Template files support variables and placeholders (date, author, item number)
- [ ] Users can select a non-default template at item creation time; `config.yaml` default is used as fallback
- [ ] Support organization-level shared templates

#### Folder layout

```
.centy/
  issues/
    config.yaml
    templates/
      default.md
      bug-report.md
      feature-request.md
    0001.md
  features/
    config.yaml
    templates/
      default.md
    feat-login.md
```

#### `config.yaml` template section example

```yaml
template: templates/default.md
templates:
  - name: default
    path: templates/default.md
  - name: bug-report
    path: templates/bug-report.md
  - name: feature-request
    path: templates/feature-request.md
```

### Custom commands per item type (#140)

- [ ] Commands are defined in each item type's `config.yaml` under a `commands:` key, scoped to that type
- [ ] Commands are surfaced through the entity actions system so UIs can discover and display them
- [ ] Supported action types: `shell` (run an arbitrary command/script) and `builtin` (apply structured updates to the item)
- [ ] Commands receive the full item context (metadata, content, project info) as environment/input
- [ ] Commands can modify the item as part of execution (update status, fields, content)
- [ ] Support a `confirm: true` flag for commands that perform destructive or irreversible operations
- [ ] Support organization-level shared commands across all projects

#### `config.yaml` commands section example

```yaml
commands:
  - name: triage
    description: Set priority and assign to team
    action: shell
    command: "./scripts/triage.sh {item_id}"
    confirm: true
  - name: escalate
    description: Escalate to high priority
    action: builtin
    updates:
      priority: 1
      status: in-progress
```
