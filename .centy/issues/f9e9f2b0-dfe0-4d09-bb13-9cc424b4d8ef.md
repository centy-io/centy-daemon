# Upgrade mdstore to 1.0.0 and adopt new comment/flatten features

## Background

mdstore has been released at **1.0.0** (up from 0.4.0 currently in use). This is a major release with two breaking changes and one significant new feature that directly affects centy-daemon.

See the [CHANGELOG](https://github.com/centy-io/mdstore/blob/main/CHANGELOG.md) for full details.

---

## Breaking changes

### 1. `custom_fields` is now flattened to top-level frontmatter

In mdstore `Frontmatter`, `custom_fields` changed from:

```rust
#[serde(default, skip_serializing_if = "HashMap::is_empty")]
pub custom_fields: HashMap<String, serde_json::Value>,
```

to:

```rust
#[serde(flatten)]
pub custom_fields: HashMap<String, serde_json::Value>,
```

**Before (0.4.0):** custom fields were stored under a `customFields:` YAML key.  
**After (1.0.0):** custom fields are flattened directly to the top level of frontmatter — e.g., `draft: true` instead of `customFields: {draft: true}`.

**Impact on centy-daemon:**
- Generic items stored via `mdstore::storage` (in `src/item/generic/storage/crud_ops.rs`) use `mdstore::Frontmatter` directly — their existing stored files with a `customFields:` key will **deserialize incorrectly** after the upgrade (those keys will be read back as unknown/empty).
- `IssueFrontmatter` (in `src/item/entities/issue/metadata/frontmatter.rs`) is a custom struct, not mdstore's `Frontmatter`, so it is unaffected structurally — but it also uses `#[serde(default, skip_serializing_if = "HashMap::is_empty")]` (non-flattened), which means issue-specific custom fields remain nested. This is fine as-is, but worth documenting.
- **Migration needed** for any existing generic-item `.md` files that have a `customFields:` block — they must be migrated to flat top-level keys, or the reconciliation logic must handle both shapes.

### 2. `generate_frontmatter` takes a new 4th `comment` parameter

The `generate_frontmatter` function signature changed:

```rust
// 0.4.0
pub fn generate_frontmatter(fm: &T, title: &str, body: &str) -> String

// 1.0.0
pub fn generate_frontmatter(fm: &T, title: &str, body: &str, comment: Option<&str>) -> String
```

The call site in `src/item/entities/issue/create/write_issue.rs:67`:

```rust
let issue_content = generate_frontmatter(frontmatter, display_title, body);
```

**will fail to compile** — must be updated.

## New feature to adopt: native frontmatter comment support

mdstore 1.0.0 adds a `comment: Option<String>` field to `Item`, `CreateOptions`, and `UpdateOptions`. The comment is preserved through all CRU operations, soft delete, restore, duplicate, and move.

Currently, centy-daemon inserts the managed-by YAML comment (`# This file is managed by Centy. Use the Centy CLI to modify it.`) manually via `format_issue_file()` in `src/utils/format.rs`, which runs `format_markdown()` + post-processes the output to inject the comment. This is fragile (e.g., it has to un-escape `\#` produced by pulldown-cmark-to-cmark).

**With 1.0.0, we can pass the comment directly through mdstore** and it will be preserved on every write, duplicate, and move — no post-processing needed.

## Required changes

### 1. Bump version in `Cargo.toml`
```toml
mdstore = "1.0.0"
```

### 2. Fix `generate_frontmatter` call in `write_issue.rs`

In `src/item/entities/issue/create/write_issue.rs`, update the call to pass the comment:

```rust
// Before
let issue_content = generate_frontmatter(frontmatter, display_title, body);
let issue_file = issues_path.join(format!("{issue_id}.md"));
fs::write(&issue_file, format_issue_file(&issue_content)).await?;

// After — pass comment directly; no format_issue_file post-processing needed
use crate::utils::format::CENTY_HEADER_YAML;
let issue_content = generate_frontmatter(frontmatter, display_title, body, Some(CENTY_HEADER_YAML));
let issue_file = issues_path.join(format!("{issue_id}.md"));
fs::write(&issue_file, &issue_content).await?;
```

### 3. Pass `comment` in `CreateOptions` and `UpdateOptions` for generic items

In `src/item/generic/storage/crud_ops.rs` (and any other place building `CreateOptions`/`UpdateOptions`), set:

```rust
CreateOptions {
    // ...existing fields...
    comment: Some(CENTY_HEADER_YAML.to_string()),
}

UpdateOptions {
    // ...existing fields...
    comment: None, // preserve existing comment (mdstore semantics: None = keep)
}
```

This ensures generic items also get the managed-by comment on creation and retain it on update.

### 4. Simplify or remove `format_issue_file` from `src/utils/format.rs`

Once the comment is passed natively through mdstore, `format_issue_file()` is no longer needed for the comment-insertion logic. The `format_markdown()` call it wraps is also unnecessary if we are not normalizing the body through pulldown-cmark before writing (mdstore stores body as-is).

Evaluate whether `format_issue_file` can be deleted entirely, or whether `format_markdown` is still needed for other callers.

### 5. Handle the `custom_fields` flatten breaking change

Decide how to handle existing generic-item files that stored custom fields under `customFields:`:

- **Option A:** Write a one-time migration in the reconciliation/init path that detects and re-flattens legacy files.
- **Option B:** Accept that existing custom-field data will be lost on the next write (only viable if custom fields on generic items are not yet used in production).

Check whether any real data uses custom fields in generic items and choose accordingly.
