# Migrate hooks from config.json to hooks.yaml

Currently hooks can be defined in two places:

1. `config.json` via the `hooks` field (the actual runtime hooks the daemon reads)
2. `.centy/hooks.yaml` (currently just a placeholder scaffold with comments — the daemon never reads it)

**Goal:** Make `hooks.yaml` the single source of truth for hooks, using an event-driven naming convention for hook patterns. Remove the `hooks` field from `config.json` and the gRPC `Config` proto message.

**Changes required in centy-daemon:**
- Add `HooksFile` wrapper struct for deserializing `hooks.yaml` (wraps existing `HookDefinition`)
- Change `load_hooks_config` to read from `hooks.yaml` via `serde_yaml` instead of `config.json`
- Remove `hooks` field from `CentyConfig` and `config.json`
- Remove `hooks` from the gRPC `Config` proto (reserve field 12)
- Update `validate_config`, `config_merge`, and proto conversion to remove hooks
- Update `HOOKS_YAML_CONTENT` template to show real examples
- Rename hook patterns to event-driven convention (see below)

**Hook pattern naming — event-driven convention:**

Patterns follow `{item_type}.{event}` where the event name reflects what happened (or is about to happen):

| Timing | Event name | Example |
|--------|-----------|---------|
| Before | `{type}.creating` | `issue.creating` |
| After  | `{type}.created`  | `issue.created`  |
| Before | `{type}.deleting` | `link.deleting`  |
| After  | `{type}.deleted`  | `link.deleted`   |
| Before | `{type}.updating` | `issue.updating` |
| After  | `{type}.updated`  | `issue.updated`  |

Wildcards: `*.created`, `issue.*`, `*.*`

**hooks.yaml format:**
```yaml
hooks:
  - pattern: "issue.creating"
    command: "echo pre-create"
  - pattern: "*.deleted"
    command: "notify.sh"
    async: true
    timeout: 10
```
