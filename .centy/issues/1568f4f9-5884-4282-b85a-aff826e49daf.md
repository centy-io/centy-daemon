# Migrate hooks from config.json to hooks.yaml

Currently hooks can be defined in two places:

1. `config.json` via the `hooks` field (the actual runtime hooks the daemon reads)
2. `.centy/hooks.yaml` (currently just a placeholder scaffold with comments — the daemon never reads it)

These also use different formats:
- `config.json`: `pattern`/`command` fields with `pre:issue:create` syntax
- `hooks.yaml` example: `event`/`run` fields with `issue.created` syntax

**Goal:** Make `hooks.yaml` the single source of truth for hooks. Remove the `hooks` field from `config.json` and the gRPC `Config` proto message.

**Changes required in centy-daemon:**
- Update `HookDefinition` struct to use `event`/`run` fields (YAML format)
- Add `HooksFile` wrapper struct for deserializing `hooks.yaml`
- Update `ParsedPattern` to parse the event format (`issue.created` → `post:issue:create`)
- Change `load_hooks_config` to read from `hooks.yaml` via `serde_yaml`
- Remove `hooks` field from `CentyConfig` and `config.json`
- Remove `hooks` from the gRPC `Config` proto (reserve field 12)
- Update `validate_config`, `config_merge`, and proto conversion to remove hooks
- Update `HOOKS_YAML_CONTENT` template to show real examples with the new format

**Event format for hooks.yaml:**
- `issue.created` → post-create hook (non-zero exit is logged as warning)
- `issue.creating` → pre-create hook (non-zero exit aborts the operation)
- `*.deleted` → post-delete for any item type
- `issue.*` → all operations/phases for issues
- `*.*` → catch-all
