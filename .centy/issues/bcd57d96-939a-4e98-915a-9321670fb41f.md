# CLI requires redundant 'centy-daemon' subcommand on every invocation

## Problem

The CLI binary is named `centy`, but because `protoc-gen-cobra` generates a subcommand named after the gRPC service (`CentyDaemon` → `centy-daemon`), all RPCs are nested one level deeper than necessary:

```
centy centy-daemon list-items
centy centy-daemon create-item --title "..."
```

The `centy-daemon` segment is redundant — the binary already implies the daemon, and there is only one service.

## Root cause

In `cli/main.go` the generated command is added as a subcommand of root:

```go
root.AddCommand(centyv1.CentyDaemonClientCommand(
    client.WithServerAddr(addr),
))
```

## Selected fix: use daemonCmd as root (Option B)

`cli/main.go` becomes:

```go
func main() {
    addr := os.Getenv("CENTY_DAEMON_ADDR")
    if addr == "" {
        addr = "127.0.0.1:50051"
    }

    root := centyv1.CentyDaemonClientCommand(client.WithServerAddr(addr))
    root.Use = "centy"
    root.Short = "CLI client for the centy daemon"
    root.Long = ""
    root.Version = version

    if err := root.Execute(); err != nil {
        fmt.Fprintln(os.Stderr, err)
        os.Exit(1)
    }
}
```

## Why not Option A (hoist children)

`cfg.BindFlags(cmd.PersistentFlags())` registers 11 persistent flags on `daemonCmd` — `--server-addr`, `--request-file`, `--request-format`, `--response-format`, `--timeout`, `--tls`, and TLS cert/key flags. Cobra resolves persistent flags by walking up the parent chain. Hoisting children off `daemonCmd` severs that chain, silently dropping all connection flags unless you also manually call `root.PersistentFlags().AddFlagSet(daemonCmd.PersistentFlags())`. That's two operations where forgetting one breaks the CLI non-obviously.

Option B requires zero flag bookkeeping — the generated command becomes root and all its persistent flags are naturally inherited.
