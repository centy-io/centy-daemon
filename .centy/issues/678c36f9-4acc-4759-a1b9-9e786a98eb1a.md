# Support custom item types and templates

Enable developers to define custom item types (epics, tasks, milestones) with fully customizable fields and templates.

## Goal

Allow projects to configure custom item types beyond the built-in issues/docs, with:

* Fully customizable fields per type
* Custom templates for each type
* Reference fields to link between types

## Design Decision

Custom item types are defined via `.centy/<item>/config.yaml` files. See issue #135 for the full config.yaml schema and design.

The daemon discovers item types by scanning `.centy/*/config.yaml` at startup. Built-in types (issue, doc) are created by `centy init` using the same config.yaml system — no special code paths.

## Example Use Cases

* **Epics** — `.centy/epics/config.yaml` — high-level items grouping multiple issues
* **Tasks** — `.centy/tasks/config.yaml` — smaller work items, possibly nested under issues
* **Milestones** — `.centy/milestones/config.yaml` — time-based groupings with deadlines
* **Bugs** — `.centy/bugs/config.yaml` — separate from issues with reproduction-specific fields

## Implementation Steps

1. Define config.yaml schema (statuses, priority, features flags, custom fields, template path)
2. Daemon scans `.centy/*/config.yaml` to build in-memory type registry at startup
3. `centy init` creates `.centy/issues/config.yaml` and `.centy/docs/config.yaml` with defaults
4. Generic storage layer reads/writes items under `.centy/<type>/`
5. Generic CRUD operations driven by config.yaml feature flags
6. Field validation based on config.yaml schema
7. gRPC API accepts string `item_type` instead of enum
8. Migration: `centy update` adds config.yaml to existing `.centy/issues/` and `.centy/docs/` folders
