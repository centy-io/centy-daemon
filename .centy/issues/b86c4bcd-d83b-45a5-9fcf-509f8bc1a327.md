# MCP: npx cache corruption causes connection failures

## Problem

When Claude Code (or any MCP client) starts and runs `npx centy-mcp`, it can fail with:

```plaintext
npm error code ENOTEMPTY
npm error syscall rename
npm error path ~/.npm-cache/_npx/<hash>/node_modules/centy-mcp
npm error dest ~/.npm-cache/_npx/<hash>/node_modules/.centy-mcp-<tmp>
```

This is a known npx race condition triggered by concurrent invocations or a partially cached install. The result is a corrupted cache entry that silently prevents the MCP from connecting. The only fix is manually deleting the cache entry.

## Additional Weaknesses

**Double network hop on cold cache:** On first run, npx downloads the npm package, then the `postinstall` script downloads the actual binary from GitHub. Two network calls, two failure points. If the GitHub release asset doesn't exist for the exact npm version, the install fails and the MCP appears to hang with no clear error.

## Suggestions

-   **For distribution:** Consider adding a retry/fallback in `scripts/install.js`, or documenting the manual cache-clear workaround (`rm -rf ~/.npm-cache/_npx/<hash>`) in the MCP README.
