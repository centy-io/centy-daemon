# Consolidate asset storage to .centy/assets/ and support markdown references

## Background

Assets currently support two storage locations:

- **Issue-specific**: `.centy/issues/assets/{issue_id}/{filename}`
- **Shared**: `.centy/assets/{filename}`
- **Legacy** (read-only): `.centy/issues/{issue_id}/assets/{filename}`

Nothing links assets from inside the issue `.md` file itself, making them invisible when reading the markdown directly (e.g. in an editor or git diff).

## Goal

1. Consolidate to a single location: `.centy/assets/`, with issue association tracked in metadata/manifest rather than directory path.
2. Allow users to embed/reference assets directly in the markdown body of issue files, e.g.:

```markdown
![screenshot](.centy/assets/screenshot.png)
```

## Scope

- Remove the `AssetScope` enum (or repurpose it)
- Update `add.rs`, `get.rs`, `list.rs`, `delete.rs`, `copy.rs` in `src/item/entities/issue/assets/`
- Update handlers in `src/server/handlers/asset_add/`, `asset_read/`, `asset_delete/`
- Remove `is_shared` distinction in proto/API or repurpose as metadata only
- Drop legacy path fallback (`.centy/issues/{issue_id}/assets/`)
- Migration: decide whether to migrate existing repos on first access or provide a migration command
- `AddAssetResponse` already returns a `path` field — ensure it is correct and stable for use as a markdown reference target
- Decide whether to auto-insert the reference into the issue body on `AddAsset`, or expose the path and let the client insert it
- If auto-insert: define insertion point (append to body, at cursor position via request field, etc.)

## Open Questions

1. Auto-insert by daemon or client-driven?
2. Should the reference path be relative to the project root or the `.centy/` directory?
