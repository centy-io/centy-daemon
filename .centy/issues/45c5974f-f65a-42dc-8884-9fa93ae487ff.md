# Lifecycle hooks: HTTP webhooks, gRPC subscriptions, history, and e2e tests

## Overview

This issue consolidates several lifecycle hooks enhancements into a single tracked effort. The hooks system (introduced in #32) supports executing bash scripts before and after item operations via configurable `phase:item_type:operation` patterns. The work below extends it with native HTTP webhook delivery, real-time gRPC streaming, persistent execution history, and comprehensive e2e test coverage.

---

## Checklist

### Native HTTP Webhooks (was #154)

- [ ] Add a dedicated `webhook` hook type alongside shell commands
- [ ] Accept URL, HTTP method, headers, and a payload template in hook config
- [ ] Support configurable retry logic (attempts, backoff) and per-hook timeout
- [ ] Enable easy integration with Slack, Discord, PagerDuty, and similar services without requiring curl-in-bash workarounds

### gRPC Subscriptions for Hook Events (was #156)

- [ ] Implement `SubscribeHookEvents` server-streaming RPC
- [ ] Stream real-time hook lifecycle events: `started`, `completed`, `failed`, `blocked`
- [ ] Support filtered subscriptions by item type, operation, and/or phase
- [ ] Allow the TUI to display live hook activity and external tools to react without polling

### Hook Execution History and Logging (was #155)

- [ ] Persist every hook invocation with: timestamp, pattern, command, exit code, stdout/stderr, duration, blocking flag
- [ ] Expose `ListHookExecutions` and `GetHookExecution` gRPC RPCs
- [ ] Surface history in the TUI and CLI for debugging and auditing failed hooks

### End-to-End Tests (was #150)

- [ ] Add `e2e/hooks.e2e.spec.ts` using the `createTempProject()` fixture and Vitest
- [ ] Hook configuration via gRPC (`UpdateConfig` / `GetConfig` round-trip, persistence in `config.json`)
- [ ] Pre-hook blocking: `exit 1` blocks operation and returns error; `exit 0` allows it; item NOT created when blocked
- [ ] Post-hook execution: sync side-effects (file creation), `success` flag correctness, async (`is_async: true`) background execution
- [ ] Pattern matching: wildcards (`*:*:delete`, `pre:*:create`, `*:issue:*`), specificity ordering, disabled hooks skipped
- [ ] Hook context: env vars (`CENTY_PHASE`, `CENTY_ITEM_TYPE`, `CENTY_OPERATION`, `CENTY_PROJECT_PATH`, `CENTY_ITEM_ID`) and stdin JSON payload
- [ ] Timeout enforcement: short-timeout hook that runs a long command is killed with appropriate error
- [ ] Cross-item-type coverage: issues, docs, PRs, users, links, assets
- [ ] Dynamic config updates: adding/disabling/removing hooks takes effect immediately for subsequent operations

---

## Implementation Notes

- Follow existing e2e patterns (`writeProjectFile`, `readProjectFile`, `projectFileExists`)
- For async post-hooks use polling with a small delay to verify eventual execution
- HTTP webhook type should reuse the existing hook config schema with a new `type` discriminant
- gRPC streaming can build on the same internal event bus used by hook execution history
