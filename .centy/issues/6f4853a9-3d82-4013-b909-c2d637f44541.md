# Story linking and delete are broken for slug-based story items

## Problem

Two operations fail silently or incorrectly for the `stories` item type (which uses slug-based identifiers rather than UUIDs):

### 1 — Linking fails silently

Both ways of creating a link between a story and another item type produce no error but also create no link:

```
# No error, no link created:
centy link story <slug> child-of epic:<N>

# No error, no link created:
centy link epic <N> parent-of story:<slug>

# No error, no link created (--link flag during create):
centy create story --link child-of:epic:<N> ...
```

The link system appears to only resolve UUID-based `targetId`/`sourceId` values. Story items use slugs as their identifier (`identifier: slug` in config.yaml), so the daemon returns LINK_TARGET_NOT_FOUND or LINK_SOURCE_NOT_FOUND — except when the `--link` flag is used during `create`, which swallows the error entirely.

### 2 — Delete reports success but does not remove the file

```
centy delete story <slug>
# Returns: { "deleted": true }
# But .centy/stories/<slug>.md still exists on disk
```

The RPC reports success but the file is not removed. This leaves ghost story files that reappear as live items in future list operations.

## Reproduction

```bash
# Create a story
centy create story --title "Test" --status ready

# Try to link it
centy link story test child-of epic:1
# Error: LINK_SOURCE_NOT_FOUND

# Try to delete it
centy delete story test --json
# Output: { "deleted": true }

# File is still there:
ls .centy/stories/test.md  # exists
```

## Root cause hypothesis

The link resolution and delete handler likely branch on `identifier == uuid` and skip or short-circuit when the identifier is a slug. The slug-based code path is either missing or returns early without performing the operation.

## Impact

- Stories cannot be linked to epics or issues — making the link system unusable for this item type
- Deleted stories remain on disk and continue to appear in `centy list stories`
- The `gap-analyze` skill cannot link newly created stories to their parent epics

## Steps to fix

1. Audit the link creation handler to ensure slug-based source and target IDs are resolved to their file path before the link record is written
2. Audit the delete handler for slug-identified item types to ensure the file is actually removed after the RPC succeeds
3. Add e2e tests covering link creation and deletion for slug-based item types
