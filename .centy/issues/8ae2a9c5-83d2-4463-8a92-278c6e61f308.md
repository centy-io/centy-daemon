# lint: enable clippy::string_slice to prevent string slicing panics

String slicing in Rust (e.g., &s\[1..5\]) panics if the indices fall on a non-UTF-8 character boundary. Even when UTF-8 validity is currently guaranteed, this invariant is fragile — changes to upstream data formats or user input could introduce multi-byte characters that break at slice boundaries.

Examples in the codebase:

* src/search/parser.rs:219 — &raw\[1..raw.len() - 1\] to strip quotes
* src/item/entities/issue/id.rs:58 — &id\[..8\]
* src/item/entities/pr/id.rs:52 — &id\[..8\]
* src/metrics.rs:46 — uuid::Uuid::new_v4().to_string()\[..8\]
* src/workspace/path.rs:53,56,110,125 — multiple truncation operations

While UUIDs and ASCII identifiers are safe today, the string_slice lint would flag these patterns and encourage character-boundary-safe alternatives.

Proposal: Add string_slice = “warn” to Cargo.toml \[lints.clippy\], then promote to deny for non-test code. Use safe alternatives like id.get(..8).unwrap_or(id) or id.chars().take(8).collect().
