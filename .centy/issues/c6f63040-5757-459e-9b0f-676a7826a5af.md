# Config architecture: UpdateConfig RPC, validation middleware, and dot-key flattening

## Overview

This issue consolidates three related config architecture improvements: adding the `UpdateConfig` RPC with color support, introducing a config validation middleware layer, and migrating config to a flat dot-separated key format (VS Code style).

Consolidates: #17, #142, #159

---

## Checklist

### UpdateConfig RPC and color support (was #17)

- [ ] Add `UpdateConfig` RPC to `proto/centy.proto`
- [ ] Add `UpdateConfigRequest` and `UpdateConfigResponse` proto messages
- [ ] Add `state_colors` map to `Config` (state name → hex color)
- [ ] Add `priority_colors` map to `Config` (priority level → hex color)
- [ ] Add `LlmConfig` proto message with fields: `auto_close_on_complete`, `update_status_on_start`, `allow_direct_edits`
- [ ] Add `state_colors: HashMap<String, String>` to `CentyConfig` in `src/config/mod.rs`
- [ ] Add `priority_colors: HashMap<String, String>` to `CentyConfig` in `src/config/mod.rs`
- [ ] Implement `update_config` RPC handler in `src/server/mod.rs`
- [ ] Add `proto_to_config` conversion function
- [ ] Add `validate_config` function enforcing: `default_state` in `allowed_states`, non-empty `allowed_states`, `priority_levels` in [1, 10], unique custom field names, enum fields have `enum_values`, colors are valid hex (`#RRGGBB` or `#RGB`)
- [ ] Set default state colors: `open` → `#10b981`, `in-progress` → `#f59e0b`, `closed` → `#6b7280`
- [ ] Set default priority colors (3 levels): `1` → `#ef4444`, `2` → `#f59e0b`, `3` → `#10b981`

### Config validation middleware (was #142)

- [ ] Create `ConfigValidationLayer` / `ConfigValidationService` Tower middleware (following `GrpcLoggingLayer` pattern in `src/grpc_logging.rs`)
- [ ] Reuse `validate_config()` from `src/server/mod.rs` inside the middleware
- [ ] Register the middleware in the server builder chain in `src/main.rs`
- [ ] Return gRPC `FAILED_PRECONDITION` status with validation error message on failure, before the request reaches the service handler
- [ ] Decide which RPCs are exempt from validation (e.g. `GetConfig`, `Shutdown`)
- [ ] Evaluate and implement config caching (file-watcher invalidation or short-TTL in-memory cache) to avoid per-request I/O overhead

### Flat dot-separated config keys (was #159)

- [ ] Update `CentyConfig` / `LlmConfig` serde model in `src/config/mod.rs` to deserialize from either nested or flat format but always serialize to flat format
- [ ] Create `src/config/migrate.rs` with conversion logic from nested to flat format
- [ ] Call migration in `read_config()` before deserializing
- [ ] Log a deprecation warning when the old nested format is detected
- [ ] Auto-convert on read: write the flat format back to disk when old nested format is found
- [ ] Extend the pattern to all nested config sections (not just `llm`)
- [ ] Update the proto/gRPC layer to ensure flat keys are properly mapped
- [ ] Add E2E tests verifying both formats are accepted and old format is auto-converted
