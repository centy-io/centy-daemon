# Remove status feature flag from config.yaml — derive from statuses list

## Problem

The `features.status` boolean in item-type `config.yaml` files is redundant. Whether status is enabled is already implied by the presence of the `statuses` list:

* `features.status: true` + non-empty `statuses` list → status is on
* `features.status: false` + no `statuses` list → status is off

Having both is duplication and a potential source of inconsistency (e.g. `status: true` but no `statuses`, or vice versa).

## Proposal

Remove the `status` key from every `features` block in `config.yaml` files. Derive the status feature at runtime:

* **on** — `statuses` list is present and non-empty
* **off** — `statuses` list is absent or empty

## Scope

1. Remove `features.status` from all existing `.centy/*/config.yaml` files.
1. Update the Rust config parser / deserialization logic to no longer read `features.status`.
1. Wherever `features.status` is checked in code, replace the check with `!statuses.is_empty()` (or equivalent).
1. Update any config-generation / template code so new item types are not given a `status` feature flag.
1. Update tests that assert on `features.status`.
