# Remove defaultStatus from item type config — first status is the default

## Summary

The `defaultStatus` field in the item type config (e.g. `.centy/issues/config.yaml`) is redundant. The convention should be: **the first entry in `statuses` is the default**.

## Motivation

* Removes an explicit config field that must be kept in sync with the `statuses` list
* Simplifies the config schema — one less thing to configure
* The “first status is default” convention is intuitive and matches common patterns

## Changes Required

### `src/config/item_type_config/types.rs`

* Remove `default_status: Option<String>` field from `ItemTypeConfig`

### `src/config/item_type_config/defaults.rs`

* Remove the validation that checks `defaultStatus` is in `statuses`
* Update `default_doc_config()` (already has `default_status: None`, just remove the field)

### `src/config/item_type_config/mod.rs`

* Remove the line that sets `issue_config.default_status = statuses.first().cloned()` (no longer needed since callers will use `statuses.first()` directly)

### `src/config/item_type_config/convert.rs`

* Remove `default_status` from conversion functions

### `src/item/entities/issue/create/create_fn.rs`

* Replace `.and_then(|c| c.default_status.clone())` with `.and_then(|c| c.statuses.first().cloned())`

### `src/server/handlers/item_type_create/`

* `build.rs`: Remove the `default_status` mapping
* `validate.rs`: Remove `default_status` validation
* `tests.rs`: Update test fixtures

### `src/server/convert_entity.rs`

* Remove `default_status` from proto conversion

### Tests

* Update all test fixtures that set `default_status` explicitly
* Update assertions accordingly
* Files: `itc_tests_1.rs`, `itc_tests_2.rs`, `itc_tests_3.rs`, `itc_tests_4.rs`, `itc_tests_8.rs`, `execute_tests_3.rs`, `item_type_config_migration_test.rs`

### Config migration / backwards-compat

* Existing `config.yaml` files that have `defaultStatus:` should be silently ignored on read (serde must not use `deny_unknown_fields`, or add a migration step)
