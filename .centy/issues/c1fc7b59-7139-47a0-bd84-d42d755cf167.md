# Remove LinkTargetType enum in favor of string-based *_item_type fields

## Background

`LinkTargetType` is a legacy protobuf enum with only 2 active values (`ISSUE`, `DOC`) and one reserved slot. All messages that use it already have parallel string-based `*_item_type` fields that supersede it, following the same pattern used to remove `EntityType`.

The internal Rust representation (`TargetType`) has always been a pure string wrapper — the enum only survives for wire-compatibility with old clients.

## Current state

The enum appears in 3 proto messages:
- `Link.target_type = 2`
- `CreateLinkRequest.source_type = 3`, `target_type = 5`
- `ListLinksRequest.entity_type = 3`

The resolver (`resolve_target_type()` in `src/server/convert_link.rs`) already treats the string field as authoritative and only falls back to the enum when the string is empty.

## What needs to change

1. **Proto** — reserve the old enum field numbers, delete the `LinkTargetType` enum definition
2. **Rust** — remove `resolve_target_type()` and `proto_link_target_to_internal()` / `internal_target_type_to_proto()`; use string fields directly in `link_create`, `link_read`, and `link_view_to_proto()`
3. **E2E tests** — replace `LinkTargetType.ISSUE` → `"issue"`, `LinkTargetType.DOC` → `"doc"`, remove the enum from `e2e/fixtures/grpc-client.ts`

## Reference

Same pattern as the `EntityType` removal (commit 0479d56).
