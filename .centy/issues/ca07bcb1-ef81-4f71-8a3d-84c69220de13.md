# link: target type prefix ignored, always resolved as source type

## Description

When using the `link` command with a cross-type target (e.g., `plan:<uuid>`), the daemon ignores the type prefix in the `TARGET` argument and resolves the UUID using the source type instead. This makes cross-type linking impossible when using full UUIDs.

## Steps to Reproduce

1. Create an issue (UUID: `aaa...`) and a plan (UUID: `bbb...`)
2. Attempt to link using full UUIDs to avoid display number collision:
   ```
   centy link issue aaa... relates-to plan:bbb...
   ```
3. Error: `Target entity not found: bbb... (issue)`

## Expected Behavior

The `type:id` format in the TARGET argument should resolve the entity using the specified type (`plan`), not the source type (`issue`).

## Actual Behavior

```json
{"message":"Target entity not found: bbb... (issue)","code":"LINK_TARGET_NOT_FOUND"}
```

Note the `(issue)` in the error — the type prefix `plan:` is being ignored and the daemon falls back to the source type for target resolution.
