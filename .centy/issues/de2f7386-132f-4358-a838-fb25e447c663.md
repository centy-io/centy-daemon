# Show linkage in all item types

## Problem

Links between items are stored in a separate normalized store (`.centy/links/`), but they are **invisible** when you retrieve, list, or search any item. To discover an item's relationships you must make a separate `GetLinks` call with the item ID — this is a significant usability gap.

Additionally, only `issue` appears as a source type in practice; epics, stories, micro-tasks, ideas, docs, assets, personas, and routines are not first-class participants in the link graph. The link system is effectively siloed.

## Current state

- `GetItem` / `ListItems` / `SearchItems` return no link data
- `centy get issue 5` shows zero relational context in output
- 23 links in `.centy/links/` — all originate from `issue`; targets are `epic` (19), `issue` (3), `doc` (1)
- Link types in use: `child-of` (20), `relates-to` (2), `blocked-by` (1)
- Understanding an item's place in the project requires N+1 round-trips

## Desired behavior

### 1. Inline links in `GetItem` responses

Every item retrieval should include a `links` field with fully resolved relationships:

```json
{
  "id": "abc123",
  "type": "issue",
  "title": "Fix auth middleware",
  "status": "open",
  "links": [
    {
      "link_type": "child-of",
      "direction": "outbound",
      "related_id": "epic-42",
      "related_type": "epic",
      "related_title": "Auth overhaul"
    },
    {
      "link_type": "blocked-by",
      "direction": "inbound",
      "related_id": "issue-7",
      "related_type": "issue",
      "related_title": "Upgrade JWT library"
    }
  ]
}
```

- **Direction**: `outbound` = this item is the source; `inbound` = this item is the target
- **Title resolution**: look up the related item's title at read time (avoid bare ID references)
- **Bidirectional**: if A `child-of` B, then B's links view shows A as an inbound `child-of`

### 2. Link summary in `ListItems` / `SearchItems`

Bulk listing should include at minimum a `link_count` per item. A `--include-links` flag can opt into full link resolution (slower but richer):

```json
{ "id": "...", "title": "...", "link_count": 3 }
```

### 3. All item types as link participants

Every item type — issues, epics, stories, micro-tasks, ideas, docs, assets, personas, routines, comments — must be linkable as both source and target. The `CreateLink` tool and `centy create ... --link` flag should enforce no type restriction.

### 4. CLI output

`centy get issue 5` should render a **Links** section:

```
Links
  → child-of  Epic #3: Auth overhaul
  ← blocked-by  Issue #7: Upgrade JWT library
```

(`→` outbound, `←` inbound)

## Acceptance criteria

- [ ] `GetItem` for any item type returns a populated `links` array (empty array if no links)
- [ ] Links include `link_type`, `direction`, `related_id`, `related_type`, `related_title`
- [ ] `ListItems` returns `link_count` per item; `--include-links` returns full link data
- [ ] All item types can appear as `sourceType` or `targetType` in a link record
- [ ] `centy get <type> <id>` renders a Links section in terminal output
- [ ] Inbound links are surfaced from the target's perspective (bidirectional traversal)
- [ ] `GetLinks` standalone tool continues to work (backward compat)
- [ ] No N+1 issue: link resolution for `ListItems` batches the links lookup

## Implementation notes

Links are normalized in `.centy/links/` — joining them onto items requires a scan of that directory filtered by `sourceId` or `targetId`. At small scale this is acceptable. At larger scale (1k+ items) a reverse index (keyed by item ID → link IDs) or embedding a `links:` array in item frontmatter could be considered, but that is a separate performance optimization issue.

Title resolution for `related_title` requires a secondary read of the referenced item file; this should be done with a batch read, not a per-link sequential read.
