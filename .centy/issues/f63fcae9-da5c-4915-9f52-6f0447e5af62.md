# Add custom `StartDaemon` (and `IsRunning`) tools to the MCP server

## Problem

The MCP currently exposes no way to start the daemon when it isn't running. Every tool in `mcp/gen/centy/v1/centyv1mcp/centy.pb.mcp.go` is auto-generated from proto and forwards calls to the daemon via gRPC — so if the daemon is down, all tools fail. There's no lifecycle escape hatch.

The CLI's `centy start` handles this by directly spawning the binary as a detached process (`spawn(daemonPath, [], { detached: true, stdio: 'ignore' })`), but that logic lives in `centy-cli` and isn't reachable from MCP clients like Claude.

## Why it's feasible

The MCP server (`mcp/main.go`) is a **separate Go binary** — it doesn't require the daemon to be running to start up. It only needs the daemon alive when forwarding gRPC calls. This means we can register a custom hand-written `StartDaemon` tool that operates at the OS level (spawn a process) rather than going through gRPC.

The `mark3labs/mcp-go` server exposes `raw.AddTool()` for exactly this — adding tools alongside the generated ones. No proto change needed.

## Proposed implementation

Add two custom tools to `mcp/main.go` (or a new `mcp/tools/` package):

### 1. `StartDaemon`

```go
raw.AddTool(mcp.NewTool("StartDaemon",
    mcp.WithDescription("Start the centy daemon if it is not already running"),
), func(ctx context.Context, req *mcp.CallToolRequest) (*mcp.CallToolResult, error) {
    // 1. Check if already running (try TCP dial on addr, fast timeout)
    // 2. If running: return "Daemon is already running"
    // 3. Find binary: $CENTY_DAEMON_PATH → ~/.centy/bin/centy-daemon → exec.LookPath("centy-daemon")
    // 4. Spawn detached:
    //      cmd := exec.Command(binaryPath)
    //      cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}
    //      cmd.Stdout, cmd.Stderr = nil, nil
    //      cmd.Start()
    //      cmd.Process.Release()
    // 5. Poll addr every 200ms for up to 5s
    // 6. Return success or timeout error
})
```

### 2. `IsRunning`

```go
raw.AddTool(mcp.NewTool("IsRunning",
    mcp.WithDescription("Check whether the centy daemon is currently running"),
), func(ctx context.Context, req *mcp.CallToolRequest) (*mcp.CallToolResult, error) {
    // TCP dial addr with short timeout, return bool + addr
})
```

`IsRunning` is useful on its own so MCP clients can cleanly check status without calling a gRPC tool and getting a confusing connection-refused error.

## Binary search order

Should mirror what the CLI does (`centy-cli/src/lib/start/find-daemon-binary.ts`):

1. `$CENTY_DAEMON_PATH` env var
2. `~/.centy/bin/centy-daemon`
3. `exec.LookPath("centy-daemon")` (PATH fallback)

## Notes

- `addr` is already resolved from `$CENTY_DAEMON_ADDR` in `main.go` — pass it through to both tools so they respect custom ports
- On Windows, `SysProcAttr` differs (`CreationFlags: syscall.CREATE_NEW_PROCESS_GROUP`) — worth a build tag or runtime check
- No proto change, no re-generation needed; these are purely MCP-layer tools
