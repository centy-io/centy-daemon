# Use snake_case in all generated YAML

Switch all YAML output (frontmatter, config files, manifest) from camelCase to snake_case for consistency with common YAML conventions and better readability.

## Current state

All serde-serialized structs that produce YAML use `#[serde(rename_all = "camelCase")]`, resulting in keys like `displayNumber`, `createdAt`, `updatedAt`, `schemaVersion`, etc.

Affected areas:

* **Issue/doc frontmatter** (`src/common/metadata.rs`, `src/item/entities/issue/metadata.rs`, `src/item/entities/doc/crud.rs`) — `displayNumber`, `createdAt`, `updatedAt`, `deletedAt`
* **Manifest** (`src/manifest/types.rs`) — `schemaVersion`, `centyVersion`, `createdAt`, `updatedAt`
* **Config** (`src/config/mod.rs`) — `defaultEditor`, `orgId`, etc.
* **Hooks config** (`src/hooks/config.rs`) — hook field names
* **Workspace metadata** (`src/workspace/metadata.rs`, `src/workspace/types.rs`) — workspace-related fields
* **User config** (`src/user/types.rs`) — user-related fields
* **Registry types** (`src/registry/types.rs`) — registry field names

## Target state

All generated YAML keys should use snake_case: `display_number`, `created_at`, `updated_at`, `schema_version`, `centy_version`, `default_editor`, `org_id`, etc.

## Migration

Since changing key names is a breaking change for existing `.centy/` directories, a migration step is needed:

1. Add a migration in `centy update` that rewrites existing frontmatter and config files from camelCase to snake_case
1. During a transition period, accept both camelCase and snake_case on deserialization (using `#[serde(alias = "...")]`)
1. Always serialize using snake_case

## Implementation

* Change all `#[serde(rename_all = "camelCase")]` to `#[serde(rename_all = "snake_case")]` on YAML-facing structs
* Add `#[serde(alias = "camelCaseName")]` for each field to maintain backward compatibility during migration
* Add a migration step to rewrite existing files
* Update tests and snapshots
* Update any documentation referencing the old key names
