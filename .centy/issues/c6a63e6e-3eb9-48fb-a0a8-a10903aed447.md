# Auto hard-delete artifacts after configurable retention period

## Summary

Add a configuration option in the project config (e.g. `config.yaml`) that specifies a retention period for soft-deleted artifacts. Once the retention period has elapsed since an artifact’s `deleted_at` timestamp, the artifact should be automatically hard-deleted (permanently removed from disk/storage).

## Motivation

Currently, deleted artifacts are soft-deleted (a `deleted_at` timestamp is set) but never cleaned up. Over time this leads to accumulation of stale deleted data. Users should be able to configure a TTL so that old deleted artifacts are purged automatically.

## Proposed Config Shape

````yaml
cleanup:
  retention_period: "30d"  # e.g. 7d, 24h, 30d — set to null or 0 to disable
````

## Behavior

* A background task (or periodic check) inspects all artifacts with a `deleted_at` value.
* If `now - deleted_at >= retention_period`, the artifact is hard-deleted.
* The default `retention_period` is `30d` when not specified.
* If `retention_period` is explicitly `null` or `0`, auto-cleanup is disabled.
* Cleanup can run at daemon start and/or on a periodic interval (e.g. every hour).

## Acceptance Criteria

* [ ] `retention_period` field added to project config schema with default of `30d`
* [ ] Config parsing handles duration strings (e.g. `7d`, `24h`, `30d`)
* [ ] Background cleanup task iterates soft-deleted artifacts and hard-deletes expired ones
* [ ] Cleanup is a no-op when `retention_period` is explicitly `null` or `0`
* [ ] Tests cover: expired artifact is hard-deleted, non-expired is kept, disabled config is a no-op
