# Generic item create/update missing managed-by header

## Problem

All item CRUD now goes through generic gRPC handlers (`CreateItem`, `UpdateItem`, `SoftDeleteItem`, `RestoreItem`, `DuplicateItem`, `MoveItem`). These delegate to `mdstore::create()` / `mdstore::update()` etc., which write `.md` files directly **without** the YAML comment:

\`\`\`
# This file is managed by Centy. Use the Centy CLI to modify it.
\`\`\`

The old issue-specific gRPC handlers that called `format_issue_file()` no longer exist. Items created/updated via the generic layer are missing this header.

## Affected operations

- `generic_create` (`mdstore::create`) — no header on create
- `generic_update` (`mdstore::update`) — no header preserved/added on update
- `generic_soft_delete` — no header on soft delete write
- `generic_restore` — no header on restore write
- `generic_duplicate` (`mdstore::duplicate`) — no header on duplicated file
- `generic_move` (`mdstore::move_item`) — no header on moved file

## Resolution

Resolved in PR #176 by upgrading to **mdstore 1.0.0**, which supports native comment injection via the `comment` field on `CreateOptions`:

```rust
CreateOptions {
    title,
    body,
    ...,
    comment: Some(CENTY_HEADER_YAML.to_string()),
}
```

The `build_options()` function in `src/server/handlers/item_create/operation.rs` sets `comment: Some(CENTY_HEADER_YAML.to_string())`, so all items created via the generic `CreateItem` handler get the managed-by header written natively by mdstore.

`UpdateOptions`, `soft_delete`, `restore`, `duplicate`, and `move_item` in mdstore 1.0.0 preserve the existing frontmatter comment, so the header is not lost on subsequent writes.

The manual post-processing approach (`ensure_frontmatter_yaml_header()`) proposed in the original fix description was **not needed** — mdstore 1.0.0 handles it at the library level.
