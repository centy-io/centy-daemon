# Migrate link storage to per-link markdown files with frontmatter

## Summary

Change how links are stored: instead of per-entity JSON files (`.centy/{type}s/links/{entity_id}/links.json`), each link becomes its own markdown file at `.centy/links/{uuid}.md` with all data in YAML frontmatter — making links consistent with how generic items are stored.

## New storage format

```
.centy/links/{uuid}.md
```

Frontmatter fields:
- `sourceId` — source entity UUID
- `sourceType` — singular item type (e.g. `issue`)
- `targetId` — target entity UUID
- `targetType` — singular item type
- `linkType` — relationship type (e.g. `blocks`), always stored from source's perspective
- `createdAt` / `updatedAt` — timestamps from mdstore

The markdown body is intentionally empty by default — users may write whatever freeform content they like there (notes, context, etc.). The daemon ignores it.

## Key decisions

- **One file per relationship** (no separate inverse file; direction synthesized at query time)
- **No display numbers**
- **Drop old format** (no backward compat migration)
- Use `mdstore::create/delete/list` with `TypeFeatures::default()` (all features off)
- **Delete by link UUID** — `delete_link` takes the link file UUID directly
- **No inverse link types** — `linkType` is always stored from the source's perspective; the link response includes a `direction` field (`source` or `target`) so callers know which side the queried entity is on. The caller derives meaning from `linkType` + `direction` (e.g. target of `blocks` = blocked). This eliminates the need for inverse name storage or lookup entirely.

## Changes required

- New `LinkRecord` struct (full both-sides record) + `source_link()` / `target_link()` view methods (no inverse param needed)
- Rewrite `storage/io.rs` to use mdstore on `.centy/links/`
- Delete `storage/links_file.rs` and `LinksFile` type
- Rewrite `crud_fns/create.rs` — one file, synthesize both response views
- Rewrite `crud_fns/delete.rs` — delete by link UUID (mdstore delete by id)
- Rewrite `crud_read.rs` — `list_links` scans all files, filters by entity, returns direction per match; no `custom_types` param needed
- Remove `inverse` field from `CustomLinkTypeDefinition`
- Remove `get_inverse_link_type` function
- Update proto `Link` message — add `direction` field (`source`/`target`), remove any inverse name field
- Update `link/mod.rs` exports (remove `LinksFile`, `read_links`, `write_links`, `get_inverse_link_type`)
- Update `server/handlers/link_read.rs` — no config fetch needed before `list_links`
- Rewrite `links_file_*` tests for new storage model
