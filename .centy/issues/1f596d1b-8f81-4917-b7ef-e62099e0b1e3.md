# Org repo tracking, discovery, and CreateItem routing

First implementation step for the org-wide `.centy` repo feature (spec: `org-wide-centy-repo`). Covers auto-tracking the org repo on init, registry-based discovery, and routing `CreateItem` writes based on the `projects` field.

## Auto-Tracking on Init

When `centy init` is run inside a directory named `.centy`, the daemon tracks it as a normal project. Its folder name is the signal — no special flag or designation is required.

## Org Repo Discovery

Implement a per-request helper that, given a `project_path`, returns the org repo's tracked project (`Option<&TrackedProject>`):

1. Determine the current project's organization slug.
2. Scan tracked projects for one in the same org whose path ends in `/.centy`.
3. Return it, or `None` if not found.

No caching. No config fields. No filesystem heuristics.

## CreateItem Routing

`CreateItem` accepts a `projects: repeated string` field. The daemon uses it to determine where to write:

- `len(projects) > 1` → organization-wide item: discover the org repo and write there. Return a clear error if no org repo is tracked.
- `len(projects) <= 1` → project item: write to the project's own `.centy/` as normal.

In both cases, `projects` is stored as-is in the item's metadata.

Data-root resolution for the org repo: for a project whose path ends in `/.centy`, the data root is the repo root itself (no nested `.centy/` subfolder).

## Acceptance Criteria

- [ ] `centy init` in a folder named `.centy` registers it as a normal tracked project
- [ ] Discovery helper finds the org repo by org slug + path suffix `/.centy`
- [ ] Data-root resolution uses the repo root directly for projects named `.centy`
- [ ] `CreateItem` with `len(projects) > 1` writes to the org repo and stores `projects` in metadata
- [ ] `CreateItem` with `len(projects) > 1` and no org repo tracked returns a clear error
- [ ] Unit tests cover discovery (found, not found), data-root resolution, and CreateItem routing
